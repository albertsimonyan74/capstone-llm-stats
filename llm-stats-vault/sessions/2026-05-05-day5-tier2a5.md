# 2026-05-05 — Tier 2A.5 visualization corrections + calibration auto-heal

## Calibration regression — root cause + fix

**Cause.** `scripts/calibration_analysis.py` writes the canonical
`experiments/results_v2/calibration.json` from per-bucket ECE/accuracy ONLY.
It does not emit the derived fields `accuracy_calibration_correlation` and
`formatting_failure_rate_per_model`, which are computed by
`scripts/recompute_downstream.py` (step 3d) and appended to the same JSON.

When Tier 2A re-ran `calibration_analysis.py` to re-theme the reliability
PNG, it silently overwrote the canonical file and dropped the two derived
fields. Frontend (`MethodologyPanels.jsx:551–552`), backend
(`v2_routes.py:421–422`) and the deployed static-data copy all silently
broke until detected.

**Fix.** Append `subprocess.run(["python", "scripts/recompute_downstream.py"],
check=True)` to the end of `calibration_analysis.py:main()`. Auto-heal:
whenever the canonical file is overwritten, derived fields are regenerated
in the same invocation. Verified by re-running the full chain and
confirming both fields present afterward.

## Workflow gotcha (record for future sessions)

Two scripts can overwrite `experiments/results_v2/calibration.json`:

1. `scripts/calibration_analysis.py` — emits per-bucket ECE/accuracy.
2. `scripts/recompute_downstream.py` — appends `accuracy_calibration_correlation`
   + `formatting_failure_rate_per_model`.

Step 1 must always be followed by step 2. The auto-chain in
calibration_analysis.py main() now enforces this — running step 1 alone
is no longer possible without also re-running step 2.

If you ever invoke calibration_analysis.py via `import` (rather than
`__main__`) and skip the chain, manually run `python
scripts/recompute_downstream.py` afterward.

### 546 runs.jsonl Records Missing confidence_score / reasoning_score

- **What:** Only 562/1108 records have `confidence_score` and `reasoning_score` fields. The 546 missing records include 491 valid (non-error) runs across all 5 models.
- **Files:** `experiments/results_v1/runs.jsonl`, `scripts/recompute_scores.py`

- **Root cause:** `scripts/recompute_scores.py` only loaded `data/benchmark_v1/tasks.json` (Phase 1 — 136 tasks), not `tasks_all.json` (171 tasks). Phase 2 runs and synthetic runs never had their scores computed.
- **Fix:** Update `recompute_scores.py` to use `data/benchmark_v1/tasks_all.json`. Re-run: `python scripts/recompute_scores.py`.

### results.json Schema Mismatch vs CLAUDE.md Documentation

- **What:** `CLAUDE.md §7` states "top-level is a **list** of aggregate objects (not a dict with `model_aggregates` key)." The actual `results.json` written by `run_benchmark.py` is a dict with keys `model_aggregates` and `task_scores`.
- **Files:** `experiments/results_v1/results.json`, `experiments/run_benchmark.py`
- **Impact:** Any analysis code written against the CLAUDE.md description (expecting a list) will fail at import. The backend `main.py` does not consume `results.json` at all — it reads `runs.jsonl` directly — so this is currently latent.
- **Fix:** Either update CLAUDE.md §7 to match actual output, or update `run_benchmark.py` to write the list format.

## Technical Debt

### Dual Scoring Paths Must Be Manually Kept in Sync

- **What:** Scoring weights (N=M=A=C=R=0.20) are duplicated in two files: `llm_runner/response_parser.py` (W_N, W_M, etc.) and `evaluation/metrics.py` (WEIGHTS dict). A comment in each file references the other, but there is no programmatic enforcement.
- **Files:** `llm_runner/response_parser.py` lines 24–28, `evaluation/metrics.py` line 31
- **Impact:** A weight change in one file that misses the other silently produces divergent scores between the runner path and the formal evaluation path.
- **Fix:** Extract weights to a single `evaluation/weights.py` module. Import from both files.

### CONCEPTUAL Task Scoring Diverges Between Paths

- **What:** `response_parser.full_score()` uses `1.0 * rubric_score` for rubric-only tasks and `0.6 * numeric + 0.4 * rubric` for mixed tasks. `evaluation/metrics.py` always uses the 5-component weighted sum (N/M/A/C/R) regardless of task type. CONCEPTUAL tasks in Phase 1 have no numeric targets, so `metrics.py` returns N=1.0 (no targets → full credit by default), M and A from keyword checks, C and R from defaults.
- **Files:** `llm_runner/response_parser.py` lines 466–471, `evaluation/metrics.py` `numerical_score()` line 122
- **Impact:** CONCEPTUAL task scores computed during live runs differ from scores computed during formal evaluation. This inconsistency is not flagged or documented beyond CLAUDE.md §7.

### _fake_flags Destroys Per-Check Granularity

- **What:** Both `experiments/run_benchmark.py` and `experiments/runs_jsonl_adapter.py` reconstruct per-check boolean flags from a single aggregate score using `_fake_flags()`. This assigns True to the first `round(score*n)` checks alphabetically — not the checks that actually passed.
- **Files:** `experiments/run_benchmark.py` lines 40–43, `experiments/runs_jsonl_adapter.py` lines 31–42
- **Impact:** Per-check diagnostic analysis (e.g., "which structure check did models fail most?") is impossible from stored runs. The aggregate score is preserved, but the check-level signal is synthetic.
- **Note:** This is a known, documented workaround. Fixing it requires storing per-check flags in `runs.jsonl` at run time.

### LLM-as-Judge Uses Anthropic SDK (Different from Runner)

- **What:** `evaluation/llm_judge.py` imports `anthropic` (vendor SDK) and hardcodes model `claude-sonnet-4-6`. All other model clients in `llm_runner/model_clients.py` use `httpx` directly (no SDKs). The runner client uses `claude-sonnet-4-5`.
- **Files:** `evaluation/llm_judge.py` lines 10–12, `llm_runner/model_clients.py` line 105
- **Impact:** Two inconsistencies: (1) an extra dependency (`anthropic` SDK) required only for judge path; (2) the judge model is a different version than the claude client, meaning judge-assisted runs are scored by a different model than what ran the task.

### Gemini Inconsistent Token Budget

- **What:** All clients except Gemini use `_MAX_TOKENS = 1024`. Gemini uses `maxOutputTokens: 4096`.
- **Files:** `llm_runner/model_clients.py` lines 31, 179
- **Impact:** Gemini responses can be 4× longer than other models, making length-based reasoning quality signals (criterion 1 in `reasoning_quality_score`) harder to compare across models.

### build_tasks_bayesian.py Is 1706 Lines With No Tests

- **What:** `baseline/bayesian/build_tasks_bayesian.py` is the largest file in the project at 1706 lines. It generates all Phase 1 ground-truth task specs. No automated tests exist for it.
- **Files:** `baseline/bayesian/build_tasks_bayesian.py`
- **Impact:** Regenerating tasks (required after any ground-truth fix) could silently corrupt task specs with no test safety net. Only `baseline/frequentist/test_frequentist.py` (24 tests) and `capstone_mcp/test_server.py` (29 tests) exist.

### Backend API Re-reads Files on Every Request (No Cache)

- **What:** `capstone-website/backend/main.py` calls `_load_tasks()` and `_load_runs()` on every API request. Both read and parse large files from disk (`tasks_all.json` = 171 tasks, `runs.jsonl` = 1108 lines).
- **Files:** `capstone-website/backend/main.py` lines 22–37, 109–115, 185–186
- **Impact:** Each frontend page load that calls `/api/tasks`, `/api/results/summary`, and `/api/leaderboard` triggers 3–6 full file reads. Acceptable for demo/development, but slow under concurrent load.
- **Fix:** Add a module-level cache with a TTL, or load once at startup and invalidate on file change.

### tasks_all.json Is a Manually-Merged Artifact With No CI Guard

- **What:** `data/benchmark_v1/tasks_all.json` (171 tasks) is regenerated by a one-liner merge command. If `tasks.json` or `tasks_advanced.json` are regenerated but `tasks_all.json` is not re-merged, the merged file silently falls behind.
- **Files:** `data/benchmark_v1/tasks_all.json`, `data/benchmark_v1/tasks.json`, `data/benchmark_v1/tasks_advanced.json`
- **Impact:** `run_benchmark.py` and the backend load `tasks_all.json`. A stale merge means some tasks are scored against wrong ground truth.
- **Fix:** Add a validation step to `build_tasks_bayesian.py` and `build_tasks_advanced.py` that auto-regenerates the merged file, or add an assertion in `run_benchmark.py` checking that `tasks_all.json` count equals `tasks.json` + `tasks_advanced.json`.

### Unknown Structure Checks Default to True (Benefit-of-Doubt)

- **What:** In `response_parser._keyword_check()`, if a check name is not present in `_STRUCTURE_KEYWORDS` or `_ASSUMPTION_KEYWORDS`, the function returns `True` (line 160). This means new task types with checks not registered in those dicts will always appear to pass structure/assumption scoring.
- **Files:** `llm_runner/response_parser.py` lines 155–162
- **Impact:** Adding a new task type with new check names without updating the keyword dicts gives that task artificially inflated M and A scores during live runs.

## Security / Data Risks

### API Keys Loaded from Environment at Construction Time

- **What:** Each client class reads its API key in `__init__` via `os.environ.get(...)`. If the key is missing, `query()` returns an error dict silently — no exception is raised, no warning is logged at startup.
- **Files:** `llm_runner/model_clients.py` lines 109, 163, 219, 275, 331
- **Impact:** A run started with a missing key produces hundreds of error records in `runs.jsonl` before the operator notices. Resume logic marks these error records as "completed" and will skip them on the next run (blocking retries).
- **Fix:** Add a startup check in `run_all_tasks.py` that validates all required API keys before starting the run loop.

### runs.jsonl Contains Raw LLM Prompts and Responses (PII Risk)

- **What:** `runs.jsonl` stores the full prompt and raw LLM response for every run (field `prompt` and `raw_response`). These fields are read and served by the FastAPI backend at `/api/results/summary` (indirectly) and `/api/task/{task_id}`.
- **Files:** `experiments/results_v1/runs.jsonl`, `capstone-website/backend/main.py`
- **Impact:** For a research benchmark this is expected, but the backend's `allow_origins=["*"]` CORS policy means any website can query these endpoints and retrieve full prompt/response text.

### FastAPI Backend Has Wildcard CORS

- **What:** `main.py` sets `allow_origins=["*"]`, `allow_methods=["*"]`, `allow_headers=["*"]`.
- **Files:** `capstone-website/backend/main.py` lines 11–15
- **Impact:** Acceptable for local development and a public demo, but any production deployment should restrict origins to the frontend domain.

## Test Coverage Gaps

### No Tests for evaluation/, baseline/bayesian/, llm_runner/

- **What:** Zero test files exist for `evaluation/metrics.py`, `evaluation/llm_judge.py`, `llm_runner/response_parser.py`, `llm_runner/model_clients.py`, `baseline/bayesian/` (all modules), or `experiments/run_benchmark.py`.
- **Files:** All modules under the above paths
- **Risk:** Scoring weight changes, keyword dict updates, or ground-truth computation changes have no regression protection.
- **Priority:** High — `response_parser.full_score()` and `metrics.score_all_models()` are the core scoring paths. Any silent regression invalidates results.

### No Tests for Schema Compatibility Between Scoring Paths

- **What:** The adapter pattern in `experiments/runs_jsonl_adapter.py` bridges two schemas. No test verifies that a run record produced by `run_all_tasks.py` can round-trip through `adapt_record()` and produce the correct `TaskRun`.
- **Files:** `experiments/runs_jsonl_adapter.py`
- **Priority:** Medium.

### recompute_scores.py Has No Idempotency Test

- **What:** `scripts/recompute_scores.py` rewrites `runs.jsonl` in place. No test verifies that running it twice produces identical output, or that it preserves error records correctly.
- **Files:** `scripts/recompute_scores.py`
- **Priority:** Medium — running it with a bug could corrupt the append-only run log.
